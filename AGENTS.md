## Page Format

### Required frontmatter (all pages)

```yaml
---
title: [Page Title]
domain: [projects | research | skills | concepts | career | learning | meta]
tags: tag-one, tag-two, tag-three
sources: [source-slug-or-url, ...]
last_updated: YYYY-MM-DD
confidence: 0.0–1.0
links: [[wiki-link-one]], [[wiki-link-two]]
---
```

### Field definitions

| Field | Rule |
|-------|------|
| `title` | Title-case noun phrase. No verbs ("Tennis Vision" not "Analyzing Tennis") |
| `domain` | Must be one of the 7 valid domains above |
| `tags` | Comma-separated, lowercase, hyphenated. 3–6 tags. No brackets. |
| `sources` | Slug or URL of every raw source used. Never empty `[]`. |
| `last_updated` | ISO date of last meaningful content change |
| `confidence` | Float 0.0–1.0. See confidence scale below. Default 0.8 for new pages. |
| `links` | `[[wiki-link]]` to related pages. Minimum 2 for project/research/concept pages. |

### Confidence scale

| Score | Meaning |
|-------|---------|
| 0.9–1.0 | Directly verified from source code / live project / personal experience |
| 0.7–0.89 | From README/docs — accurate but not depth-verified |
| 0.5–0.69 | Synthesized from secondary sources — treat as approximate |
| < 0.5 | Speculative or outdated — flag for re-verification |

### Staleness thresholds

| Domain | Re-verify after |
|--------|----------------|
| `projects` | 90 days (projects get new commits, stars change) |
| `research` | 180 days (implementations stable) |
| `skills` | 180 days |
| `concepts` | 365 days (math doesn't change) |
| `career` | 60 days |
| `learning` | 90 days |
| `meta` | 30 days |

## Operations

### Ingest

**Trigger:** `workflow_dispatch` on `agent-ingest.yml` (provide repo URL + domain).
**Script:** `_agents/scripts/auto_ingest.py`

**Step-by-step:**
1. Fetch GitHub README via API
2. Generate wiki page via Gemini API with strict schema constraints
3. **Deduplication check** — before writing, check `wiki/<domain>/<slug>.md` does not already exist. If it does, skip and log `[SKIP duplicate]`.
4. Write `.md` file to `wiki/<domain>/`
5. Update `landing/script.js`, `landing/index.html`, `wiki/index.md` via inject markers
6. Append entry to `wiki/log.md`
7. Open Pull Request for human review

**Quality gate — FAIL ingest if:**
- Frontmatter missing any required field
- `links:` is empty `[]` (auto-ingested pages must link to at least 1 existing skill or concept page)
- Body contains fenced code blocks (` ``` `) for architecture diagrams or tech stacks
- Body is shorter than 300 characters after stripping frontmatter
- `sources:` contains a URL not matching the provided repo URL
- Any hallucinated contact info (emails, phone numbers not matching `tomarharsh28303@gmail.com`)

**After merge:** run `python _agents/scripts/generate_exports.py` to refresh `llms.txt` / `graph.json`. (This runs automatically in `deploy-pages.yml`.)

**Manual ingest (Claude Code):** Use `/ingest <github-url>` — see `_agents/workflows/ingest.md`.

---

### Query

**Trigger:** Conversational — user asks a question about Harsh's knowledge graph.

**Step-by-step:**
1. Read `wiki/llms.txt` — identify the 3–5 most relevant page IDs
2. Read those pages in full
3. Synthesize an answer with citations (cite as `[[page-name]]`)
4. **If the answer is substantive** (>200 words, contains non-obvious synthesis): file it back as a new `wiki/meta/` or `wiki/concepts/` page
5. Append to `wiki/log.md`: `## [DATE] query | [question summary]`

**Query output formats:**
- Markdown page (default)
- Comparison table (for "X vs Y" questions)
- Step-by-step guide (for "how to" questions)

### Lint

**Trigger:** Every push to `wiki/**` (CI) + weekly Sunday 00:00 UTC.
**Scripts:** `validate_wiki.py` + `check_orphans.py`

**Formal lint checklist:**

| Check | Script | Failure mode |
|-------|--------|-------------|
| Required frontmatter fields present | `validate_wiki.py` | CI error (blocks merge) |
| Valid domain value | `validate_wiki.py` | CI error |
| Body ≥ 100 chars | `validate_wiki.py` | CI warning |
| At least one `[[wikilink]]` in body | `validate_wiki.py` | CI warning |
| Zero orphan pages | `check_orphans.py` | CI warning (non-blocking) |
| Zero broken `[[wikilinks]]` | `check_orphans.py` | CI warning (non-blocking) |
| No duplicate log entries | manual / log review | Flag in lint run |
| Confidence field present | manual | Flag pages with default 0.8 older than staleness threshold |

**Lint does NOT currently check (future work):**
- Contradictions between pages
- Stale pages past their `last_updated` threshold
- Hallucinated contact info or metrics
- Code block violations

## Deduplication Contract

Before ingesting any source, an agent MUST:

1. Compute `slug = repo_name.lower()`
2. Check if `wiki/<domain>/<slug>.md` exists
3. If yes → log `[SKIP] <slug> already ingested` and exit without writing
4. If no → proceed with ingest

For manual/conversational ingests: search `wiki/llms.txt` for the repo name before creating a new page. If a match is found, update the existing page instead of creating a new one.

## Style Guidelines

| Rule | Example |
|------|---------|
| Terse, precise, technical | "95% accuracy (mAP@0.5)" not "strong performance" |
| Real numbers, not vague claims | "★27 stars" not "popular repo" |
| No filler phrases | ~~"This project demonstrates"~~ |
| Headers = noun phrases | "Architecture" not "How it works" |
| Use tables for comparisons | Model variants, performance breakdowns |
| Use bullet lists for tech stacks | Categorized by layer |
| Zero fenced code blocks for architecture | Use bullet lists, blockquotes, tables |
| `[[wikilinks]]` for cross-references | Always, never bare text for page names |
| Confidence score reflects source depth | Don't default-0.8 everything |

## MCP Server (A5)

The Parallax wiki is also exposed as an MCP (Model Context Protocol) server — callable from Claude Desktop, Claude Code, or any MCP-compatible client.

**Files:** `mcp/` directory — `server.py`, `requirements.txt`, `claude_desktop_config.json`
**Dependency:** `pip install -r mcp/requirements.txt`

### Tools exposed

| Tool | Signature | Description |
|------|-----------|-------------|
| `list_pages` | `(domain?: str)` | List all pages, optionally filtered by domain |
| `get_page` | `(slug: str)` | Return full markdown of a page by slug |
| `get_related` | `(slug: str, hops?: int)` | Return slugs reachable within N hops via links graph (max 3) |
| `search_wiki` | `(query: str)` | Keyword search across all pages, returns top 10 with excerpt |

### Claude Desktop setup

Add to `~/Library/Application Support/Claude/claude_desktop_config.json` (macOS)
or `%APPDATA%\Claude\claude_desktop_config.json` (Windows):

Copy `mcp/claude_desktop_config.json` into your Claude Desktop config file
(`%APPDATA%\Claude\claude_desktop_config.json` on Windows,
`~/Library/Application Support/Claude/claude_desktop_config.json` on macOS).

### Claude Code setup

```bash
claude mcp add parallax -- python D:/LLM-wiki/mcp/server.py
```

### Design notes

- No vector DB — keyword search ranks by raw hit count (consistent with zero-dependency philosophy)
- Pages loaded once at startup; restart the server to pick up new wiki pages
- `get_related` does BFS on plain-slug frontmatter `links:` arrays (same format used by check_orphans.py)
- Slug matching is fuzzy on `get_page`: exact match first, then substring if unique
